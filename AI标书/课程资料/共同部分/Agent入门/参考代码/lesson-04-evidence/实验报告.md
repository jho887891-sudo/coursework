# Lesson 4 实验报告：三种检索策略对比

> **原则**：不使用"Agentic 更智能"作为证据。每个结论都必须回到具体数据和失败样本。

---

## 1. 实验设置

| 项目 | 配置 |
|---|---|
| 分词器 | 字符 Bigram + Unigram（确定性，无外部词典） |
| 评分函数 | TF 关键词重叠率（`matched / total`） |
| Top-K | K=3 |
| 核验阈值 | ≥0.8 → Supported，>0.5 → Partial，≤0.5 → Insufficient |
| 评测集 | 20 条 query（16 可回答 + 4 不可回答） |
| 语料库 | 10 条 passages（4 个来源、2 个版本） |
| 策略 | AlwaysRetrieve / RuleRouter / AgenticRetrieve（回退=Always） |

---

## 2. 六指标对比

| 指标 | AlwaysRetrieve | RuleRouter | AgenticRetrieve |
|---|---|---|---|
| Answer Accuracy | **100.00%** | 75.00% | **100.00%** |
| Recall@3 | **100.00%** | 75.00% | **100.00%** |
| Citation Correctness | 36.67% | 25.00% | 36.67% |
| Abstention Accuracy | 0.00% | **25.00%** | 0.00% |
| Avg Searches | 1.00 | **0.75** | 1.00 |
| Avg Latency (μs) | 113 | **95** | 280 |

### 解读

- **AlwaysRetrieve** 在所有可回答 query 上找到了证据（Recall=100%），但也对所有不可回答 query 错误声称有证据（AbstAcc=0%）。Citation Correctness 仅 36.67%，说明 Top-3 中约 2/3 的检索结果不相关——但系统未做相关性过滤就直接用于 verify。
- **RuleRouter** 节省了 25% 的搜索，但也漏掉了 3 条可回答 query（见 §3）。它的 AbstAcc 略高（25%）只是因为它碰巧跳过了部分不可回答 query，而不是因为"更聪明"。
- **AgenticRetrieve** 当前回退为 Always，结果与 Always 完全一致。没有真实模型参与决策时，"Agentic"二字不带来任何差异。

---

## 3. 三个失败样本

### 3.1 检索失败：关键词匹配的边界

RuleRouter 正确跳过了"进口设备关税优惠"（不可回答），但也跳过了以下可回答 query：

| 跳过的 query | 是否可回答 | 缺失的触发词 |
|---|---|---|
| 迟交材料如何处理 | ✅ | 不含"依据/规则/条款/比例/期限" |
| 低于成本报价需要提供什么 | ✅ | 不含规则触发词 |
| 迟交材料的后果是什么 | ✅ | 不含规则触发词 |
| 成本说明需要哪些材料 | ✅ | 不含规则触发词 |

**根本原因**：自然语言提问不一定包含法规引用关键词。一个想知道"后果"的用户不会说"条款"。

**改进方向**：触发词表需要覆盖提问句式（"如何/什么/怎么/后果/条件"），但不能无限扩大（否则退化为 Always）。

### 3.2 推理失败：主题相关 ≠ 逻辑支持

`find_failure_samples` 在当前数据集上未找到符合严格定义的推理失败样本（可回答 + Recall>0 + Insufficient），因为检索到的 passages 要么触发了 Supported/Partial（token 重叠），要么触发了 Conflicting（多源矛盾）。

但 Citation Correctness=36.67% 暴露了一个更深层的问题：**系统把 2/3 的不相关段落当作"证据"呈报**。例如 query "付款义务何时履行"检索到 REG-2024#§5.1（"合同签订后应按约定履行付款义务"），这段确实支持结论——但同一次检索的另外两条命中可能完全不相关，却也被写入了 citations。

这说明"检索到"≠"可以作为证据"，而我们的 verify() 只做 token 重叠判断，不做相关性二次筛选。

### 3.3 引用失败：应拒答而未拒答

4 条不可回答 query 的 verify 结果：

| Query | Status | 原因 |
|---|---|---|
| 履约保证金的数额 | Partial | "保证金"命中 REG-2024#§3.2 |
| 进口设备关税优惠 | Insufficient | 无 token 重叠 |
| 联合体投标的资质认定 | Partial | "投标"命中多处 |
| 什么样的条件属于不合理条件 | Partial | "不合理条件"命中 REG-2024#§7.3 |

4 条中仅 1 条正确拒答（AbstAcc=25%）。其余 3 条均因与语料共享一两个主题词而被错误判定为 Partial。

**根本原因**：关键词 baseline 无法区分"主题重叠"和"证据支持"。`履约保证金的数额` 与 `投标保证金不得超过2%` 的唯一联系是"保证金"一词——主题相关，但完全不支持问题。

**这是本课最重要的发现**：Recall 高不能证明回答正确。没有拒答能力的 RAG 系统会把任何问题都"找到依据"，这正是 Lecture 警告的"把摘要冒充法规原文"。

## 4. RuleRouter 漏检分析

```
RuleRouter 跳过的 5 条 query:
  - "迟交材料如何处理"        ⚠ 漏检（可回答）
  - "低于成本报价需要提供什么"  ⚠ 漏检（可回答）
  - "进口设备关税优惠"             （不可回答，跳过正确）
  - "迟交材料的后果是什么"      ⚠ 漏检（可回答）
  - "成本说明需要哪些材料"      ⚠ 漏检（可回答）
```

RuleRouter 的触发词表：

```rust
["依据", "规则", "条款", "比例", "期限", "保证金", "资格", "条件",
 "标准", "要求", "法规", "规定", "限额", "上限", "资质", "程序",
 "评审", "投标", "合同", "付款", "履约", "违约",
 "证据", "原文", "出处", "根据", "适用"]
```

缺失的关键词：**"如何""什么""怎么""后果""处理""成本""材料""需要""补交"**。这些都是自然语言提问的高频词，但不在当前触发词表中。补充它们会提升 Recall，但同时会缩小与 Always 的搜索次数差距——这是 Rule Router 固有的 precision-recall tradeoff。

## 5. 策略适用场景判断

| 场景 | 推荐策略 | 理由 |
|---|---|---|
| 已知法规条文检索 | RuleRouter | 用户会用"第几条""保证金""比例"等术语表达，触发词命中率高 |
| 客服/答疑系统 | AlwaysRetrieve | 用户用自然语言提问，RuleRouter 漏检风险不可接受 |
| 语料极少（<50 段） | AlwaysRetrieve | 搜索成本可忽略，RuleRouter 带来的节省不值得 Recall 损失 |
| 语料极大（>10 万段） | RuleRouter + rerank | 降低首次检索成本，但需要第二步 rerank 弥补 Recall |
| 有真实模型参与 | AgenticRetrieve | 模型可根据已获证据判断是否需要继续检索，而非"搜一次就停" |

---

## 6. 本实验的局限

1. **20 条 query 太小**：统计显著性不足，结论仅供教学讨论。
2. **Keyword-only baseline**：没有实现 BM25、embedding 或 hybrid search，Citation Correctness 低是预期内的。
3. **verify() 的阈值是手工设定的**：0.5 / 0.8 没有经过验证集调参，不同阈值下结论可能不同。
4. **AgenticRetrieve 没有真实模型**：当前回退为 Always，无法体现模型决策的价值和风险。
5. **没有衡量"过度拒答"**：AbstAcc 只看不可回答 query 的拒答率，没有检查系统是否错误拒答了可回答 query。

---

## 7. 结论

1. **Recall@3=100% 不等于系统可靠。** AlwaysRetrieve 的 Recall 满分与 AbstAcc=0% 同时成立——它找到了一切，也伪造了一切。
2. **RuleRouter 的节省来自漏检。** 搜索次数减少 25% 的同时 Recall 下降 25%，不是免费午餐。
3. **Agentic 不是魔法。** 把策略名从 "Always" 改成 "Agentic" 不改变任何指标。真实模型的价值在于"知道什么时候不该搜"，而不是"总是搜得更准"。
4. **拒答是能力，不是失败。** 一个对所有问题都声称"找到了依据"的系统，比一个敢说"我不知道"的系统更危险。
