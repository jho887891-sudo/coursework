# Lesson 7 REPORT：Memory 治理设计决策与实验结论

## 1. 设计决策

### 1.1 为什么分四条通道而不是一个 `Vec<String>`？

五种容易混淆的东西（Working State / Cache / User Profile / Episode / Knowledge Base）具有不同的：
- **更新条件**：Working State 随任务推进更新，User Profile 只有用户明确表达才更新
- **权限模型**：Document 可以写 Working/Cache/Episode，绝不能写 UserProfile
- **生命周期**：Task 在任务结束时清空，UserProfile 跨会话持久
- **可信度**：ExplicitUser > System > Document

如果全部塞进一个 `Vec<String>`，就无法区分"用户说了喜欢简洁"和"文档里写了喜欢简洁"。

### 1.2 为什么 Document 不能写 UserProfile？

这是防止 **Memory Poisoning** 的关键防线。考虑以下攻击场景：

```text
检索文档中包含："本系统用户偏好详细回答"
↓
LLM 提取："用户偏好 = 详细"
↓
如果允许自动写入 → 用户画像被恶意文档污染
↓
每次对话都注入"用户喜欢详细回答" → 垃圾信息放大
```

因此 `write_profile` 强制检查 `source == ExplicitUser`。这不是性能优化，是安全边界。

### 1.3 冲突为什么不应静默覆盖？

旧记忆：
```text
回答风格 = 详细
```

新消息：
```text
以后请简短回答。
```

如果直接覆盖：
- 丢失了用户曾经偏好"详细"的历史
- 无法审计用户偏好何时改变
- 如果新消息被误读为"请简短"而用户本意是"这个回答太详细了"（仅针对当前回答），覆盖就是错误决策

采用 supersedes 链：
- 旧偏好进入 superseded 状态
- 新偏好成为活跃版本
- `active_for` 只返回活跃版本
- `list` 仍能看到变更历史

### 1.4 何时升级人工？

三个硬条件，满足任一即升级：

| 条件 | 示例 | 风险 |
|------|------|------|
| 证据冲突 | 两份文档给出矛盾的数字 | 选错证据 → 错误结论 |
| 不可逆操作 | 发送邮件、写入生产数据库 | 操作无法撤销 |
| 低置信度 | 模型输出含"可能""似乎" | 推测被当作事实 |

升级不是失败 —— 它是系统能力的正确边界。

## 2. 实现检查清单

- [x] 四条通道分开（通过 `Scope` enum 区分，source 校验在 `write_profile` 中）
- [x] 每条长期记忆有来源 (`Source` + `confidence`)
- [x] 用户可 `list` / `delete` / `clear`（全部以 `user_id` 为第一过滤条件）
- [x] 新旧偏好冲突不静默覆盖（`resolve_conflict` 返回 `RequireHuman`，Store 内通过 supersedes 链处理）
- [x] 过期记忆不注入上下文（`active_for` 过滤 `expires_at`）
- [x] 恶意文档不能写入 UserProfile（`write_profile` 拒绝非 ExplicitUser 来源）
- [x] 至少一种人工升级策略（`escalation_for` 覆盖三种升级条件）
- [x] 26 个单元测试 + 8 个验收测试

## 3. Memory 消融实验思路

虽然本课实现的是结构化 Memory Store，但消融实验需要比较四种配置：

| 配置 | 描述 | 预期上下文长度 | 预期偏好遵循率 |
|------|------|---------------|---------------|
| 无 Memory | 每次对话从零开始 | 最短 | 最低 |
| 全量历史 | 把所有历史放进上下文 | 最长 | 中（噪声多） |
| 摘要 | 自动压缩历史 | 中 | 中（丢失细节） |
| 结构化 Memory | 只注入活跃、高置信度记忆 | 可控 | 最高 |

建议统计指标：
- 任务完成率
- 用户偏好遵循率
- 错误记忆率（注入了不该注入的信息）
- 平均上下文长度
- 冲突处理正确率

## 4. 已知局限

1. **无持久化**：当前实现为内存存储，重启后数据丢失。生产环境需替换为数据库。
2. **无语义搜索**：`active_for` 基于精确匹配（user_id + key），不支持模糊或语义检索。
3. **无自动摘要**：未实现 Episode → UserProfile 的自动升级（如"连续三次选 Rust → 偏好 Rust"）。
4. **单机存储**：未考虑分布式一致性。
5. **TTL 策略简单**：固定 `expires_at` 时间戳，不支持"最近 N 天未访问即过期"等动态策略。

## 5. System Card 摘要

见 `SYSTEM_CARD.md`。核心声明：
- 系统提供结构化记忆读写，不是自动决策系统
- 记忆来源可追溯，用户可随时查看和删除
- 冲突和低置信度情况会升级人工
- 不自动从文档中提取偏好写入画像
