## 核心概念

### 1. System Prompt 的四层信息架构

System Prompt 不是一段连续的文本。它有结构——四层，每层有明确的功能和最佳位置：

```
┌───────────────────────────────────────────────────┐
│ Layer 1: 角色定义 (Role)          — 放最前面        │
│ "你是一个招标文件合规审核专家，                         │
│  专门审查建筑工程招标文件中的排斥性条款和合规风险。"          │
│                                                     │
│ 为什么放最前面：角色定义是"注意力锚"——后续所有信息         │
│  都在"审核专家"这个 framework 下被解读。                   │
├───────────────────────────────────────────────────┤
│ Layer 2: 领域知识 (Domain)        — 放中间          │
│ "审查时关注以下维度：                                   │
│  1. 资质要求——是否明确施工总承包等级？                     │
│  2. 业绩要求——是否设置了超出项目需求的业绩门槛？             │
│  3. 报价要求——是否存在隐性价格限制？                       │
│  ..."                                                │
│                                                     │
│ 为什么放中间：领域规则是"检查清单"——LLM 需要在审查          │
│  具体条款时反复回溯这些规则。但这层最容易被"遗忘"。          │
├───────────────────────────────────────────────────┤
│ Layer 3: 输出格式 (Output)        — 放最后（靠近生成位置）│
│ "每次输出严格使用以下 JSON 格式：                         │
│  { \"finding_id\": \"...\", \"severity\": \"critical\"  │
│    |\"warning\"|\"info\", \"clause_id\": \"...\",        │
│    \"law_ref\": \"《...》第...条\", ... }"               │
│                                                     │
│ 为什么放最后：这是 LLM 生成第一个 token 之前"最后看到"     │
│  的指令——对输出格式的约束力最强。                          │
├───────────────────────────────────────────────────┤
│ Layer 4: 硬性约束 (Constraints)  — 穿插在角色和规则层     │
│ "绝对不要编造法律条款。如果找不到确切的法规依据，              │
│  在 law_ref 字段写 null，不要猜测。"                      │
│                                                     │
│ 为什么穿插：硬性约束需要在"相关上下文"中出现——               │
│  "编造法规"的禁令应该紧挨"引用法规"的指令，形成逻辑绑定。     │
└───────────────────────────────────────────────────┘
```

### 2. Lost in the Middle — LLM 注意力的 U 型曲线

#### Liu et al. (2023) 的核心发现

把同一段关键指令放在 System Prompt 的开头、中间、结尾三个位置，测试 LLM 能否正确提取：

```
位置          信息提取准确率
────          ────────────
开头 (前 10%)   ████████████████ 78%
中间 (40-60%)   ██████          32%  ← Lost!
结尾 (后 10%)   ███████████████  74%

注意力的 U 型分布：
  ↑
  │  ████              ████
  │  ████              ████
  │  ████    ████      ████
  │  ████    ████      ████
  │  ████    ████      ████
  │  ████    ████      ████
  └──┬───────┬────────┬──→ 上下文位置
    开头     中间      结尾
```

原因：Transformer 的 Position Embedding 在长序列中退化——中间位置的 token 的注意力权重受到 "位置平滑化" 影响，导致信息分辨力下降。

#### 你必须亲手做的实验

```rust
fn lost_in_middle_experiment() {
    let critical_instruction = "你的输出必须在 <answer> 和 </answer> 标签之间";
    
    // 填充文本（无意义但长度可控）
    let filler = "这是一个测试段落。".repeat(n);
    
    let positions = vec![
        ("开头", format!("{}\n{}", critical_instruction, filler)),
        ("中间", format!("{}\n{}\n{}", filler[..len/2], critical_instruction, filler[len/2..])),
        ("结尾", format!("{}\n{}", filler, critical_instruction)),
    ];
    
    for (label, prompt) in positions {
        let compliance_rate = run_50_times_and_check_if_output_has_xml_tags(prompt);
        println!("{label}: {compliance_rate:.2}%");
    }
    // 预计：开头 ~75%, 中间 ~30%, 结尾 ~70%
}
```

#### 对 Prompt 工程的直接指导

1. **最重要的指令放最前面**（角色定义 + 硬性约束）
2. **输出格式放最后面**（LLM 生成时注意力集中在最后几个 token）
3. **领域知识放中间**（可以被 Lost，但不能被删——后面用 Few-shot 示例补充）
4. **中间位置的指令如果很重要，在末尾再次重复**（"正如开头所说，你必须..."）

### 3. 标书审核 Agent 的 System Prompt 拆解

以 G4 组现有 JudgeAgent 的 System Prompt 为例（精简版），逐层分析：

```
┌─ Layer 1: 角色 ─────────────────────────────────┐
│ "你是一个专业的招标文件审核系统的核心审查员。"               │
│ 评价：太泛。"核心审查员"没有说明审查什么。                    │
│ 改进："你是标书合规审核引擎的 JudgeAgent，负责判定             │
│        招标文件条款是否存在排斥性、歧视性或不合规风险。"         │
├─ Layer 2: 领域知识 ───────────────────────────────│
│ "请从资质要求、业绩要求、报价要求、工期要求、                     │
│   安全要求、环保要求六个维度进行审查。"                         │
│ 评价：维度列出了但未定义每个维度的审查标准。                     │
│ 改进：每个维度加 1-2 句"检查什么"——"资质要求：                  │
│        检查资质等级是否超出项目规模合理需要、                     │
│        资质专业是否与项目类型匹配"                              │
├─ Layer 3: 输出格式 ───────────────────────────────│
│ "使用 search_knowledge 工具查找法规依据，                        │
│   使用 output_finding 工具输出审查结论。"                       │
│ 评价：说了用什么工具，没规定 output_finding 的参数格式。         │
│ 改进：直接给出 output_finding 的 JSON Schema 示例              │
├─ Layer 4: 硬性约束 ───────────────────────────────│
│ "每条结论必须引用具体的法规条款编号。"                            │
│ 评价：OK，但缺少"找不到时怎么办"的指导。                         │
│ 改进："如果 search_knowledge 返回的结果中找不到                    │
│        针对该条款的法规依据，在 law_ref 字段写 null                │
│        并在 suggestion 中说明'未找到相关法规'——不要猜测。"       │
└──────────────────────────────────────────────────┘
```

## 动手

### 任务 1：设计一个完整的 Agent System Prompt

为"资质要求审查 Agent"写一份 System Prompt，严格遵循四层模型。要求：
- Layer 1: 角色定义 ≤ 3 句话
- Layer 2: 领域知识 ≥ 3 个审查维度，每个维度有 1-2 句检查标准
- Layer 3: 输出 JSON 格式（含字段说明）
- Layer 4: 至少 3 条硬性约束（含"不确定时怎么做"）

### 任务 2：Lost in the Middle 验证实验

用你自己的 System Prompt 做位置置换实验。把 Layer 2 的一条关键规则（如"检查资质等级是否超出项目合理需要"）分别放在开头/中间/结尾，统计 30 次推理中这条规则被遵循的比例。

### 任务 3：拆解 G4 组现有 Prompt

选 G4 组 2 个 Agent 的 System Prompt，用四层模型逐段标注：哪段属于哪一层？哪层缺失？哪段是冗余？

## 思考题

1. "放在最前面"和"放在最后面"的指令，如果互相冲突——LLM 倾向于遵循哪个？为什么？（提示：结合 Day 1 的自回归循环——生成时注意力集中在哪个位置？）
2. 如果 System Prompt 需要同时用中文和英文（如"JSON Schema 用英文字段名"），两种语言放在一起会不会影响注意力分布？
3. Lost in the Middle 对多 Agent 对话有什么影响？如果 Agent A 的输出（放在上下文中间）被 Agent B 的 prompt 忽略了呢？

## 与标书审核项目的关系

G5 组的核心交付物之一就是 11 个 Agent 的 System Prompt `.md` 文件。你今天学会的四层模型和位置策略，直接决定了 Agent 能否正确遵循审查指令。

当你从 G4 的评测报告中看到"Agent X 在资质维度 F1=0.62"——你的第一个动作不是加更多指令，而是检查：Layer 2 的审查标准是否足够具体？输出格式是否约束了 law_ref 字段？关键约束是否被放在了中间位置（被 Lost）？
