# Day 1：规则建模方法论 — 从法条到 YAML

> "采购人不得以不合理的条件对供应商实行差别待遇"——这是《政府采购法》第 22 条的一句话。你怎么把它变成一条 Rust 代码能执行的规则？今天你学会"六要素建模"——每条规则不只是"找个关键词"，而是一个完整的、可追溯的、有适用条件的审核单元。

---

## 学习目标

1. 理解规则的六要素模型（id / category / industry / severity / source / conditions / patterns / check / suggestion）
2. 从法条原文出发，手写 10 条 YAML 规则
3. 设计行业分类体系——决定规则何时触发

---

## 核心概念

### 1. 为什么规则引擎是系统的第一道防线

假设审查一份 100 页的招标文件，Agent 使用 LLM 逐条审查需要 ~30 秒，消耗 ~5000 tokens。

但其中 90% 的问题是**确定性**的——"投标截止日期距招标文件发出日不足 20 日"、"未明确资质等级"、"报价超过预算上限"。这些问题不需要 AI——正则+关键词匹配在 50ms 内就能发现。

**规则引擎 + Agent 的分工**：

| | 规则引擎（G2） | Agent（G4+G5） |
|---|---|---|
| 处理什么 | 确定性、可模式化的问题 | 语义复杂、需要推理的问题 |
| 速度 | < 50ms / 百页文档 | ~30s |
| 成本 | 零 API 调用 | 数千 tokens |
| 可复现 | 100% 可复现 | 有随机性（Day 1 Prompt 课已讲） |
| 误报策略 | 宁可误报，不可漏检 | 精确度优先 |

### 2. 规则六要素——每条规则的本质

```yaml
rules:
  - id: "DISC-001"                    # ① 唯一ID
    category: "排斥性条款"             # ② 问题分类
    industry: "建筑工程"               # ③ 行业（决定何时触发）
    severity: "critical"               # ④ 严重程度

    # ⑤ 来源追溯——这条规则的法律依据
    source:
      law: "中华人民共和国招标投标法"
      article: "第十八条"
      version: "2017年修正版"
      effective_date: "2017-12-28"
      excerpt: "招标人不得以不合理的条件限制或者排斥潜在投标人..."

    # ⑥ 适用条件——不是全局扫描！
    conditions:
      document_type: "招标文件"
      project_types: ["施工招标", "工程总承包"]
      trigger:
        chapter_keywords: ["投标人资格", "投标人须知"]
      exclude:
        project_types: ["国际招标"]   # 国际招标可能有特殊规则

    # ⑦ 匹配模式——怎么发现
    patterns:
      - type: "regex"
        value: "(投标人|潜在投标人).*(注册地|设立.*机构).*(本市|本省|本地|辖区)"
        target: "all_clauses"
        description: "匹配地域限制类表述"
      - type: "keyword"
        value: ["排斥", "限制", "差别待遇", "歧视"]
        target: "all_clauses"
        operator: "OR"

    check: "any_match"                 # ⑧ 判定逻辑：任一模式匹配即触发
    suggestion: "该条款存在地域限制，可能违反《招标投标法》第十八条。建议删除地域性要求或改为'投标人在中标后应在项目所在地设立项目机构'。"
    law_ref: "《中华人民共和国招标投标法》第十八条"
```

**每个字段为什么存在**：

- **id**：下游引用（Agent 的输出可以说"此问题由规则 DISC-001 检测"）
- **industry**：防止误报。建筑工程需要"施工总承包资质"，IT 项目不需要——不加 industry 的规则会对 IT 项目产生误报
- **source**：可追溯性。客户问"你凭什么说这条有问题？"——指向具体法条
- **conditions**：精准触发。不是对所有文档全局扫描——只在满足触发条件时运行
- **patterns**：多模式组合。有的表述是"应具备...资质"（正面要求），有的是"不得要求..."（负面排斥）。关键词"排斥"能命中后者但可能漏掉前者——所以需要多模式互补

### 3. 从法条到规则——反向映射

```
法条原文：
  "招标人不得以不合理的条件限制或者排斥潜在投标人，
   不得对潜在投标人实行歧视待遇。"

规则建模（反向思维）：
  法条说的是 → 招标人"不得"做什么
  规则要检测的 → 文档中是否存在"做了"的证据

  patterns 要捕获的表述：
    ✅ "投标人须在本市注册成立三年以上"         → 地域限制
    ✅ "本省同类项目业绩不少于 3 个"             → 地域限制
    ✅ "投标人注册资本不低于 1 亿元人民币"       → 注册资本门槛
    ❌ "投标人须具备建筑工程施工总承包二级资质"   → 这不是排斥性——是正常的资质要求

  关键区分：资质要求 vs 排斥性条款
    资质要求 = 与项目执行能力相关的（"有施工资质才能干活"）→ 正常
    排斥性条款 = 与项目执行能力无关的（"在本市注册"才能干活？）→ 违规
```

### 4. 行业分类体系

```
建筑工程
├── 施工招标     ← "施工总承包资质"相关规则只对这个子类触发
├── 勘察招标
├── 设计招标
└── 监理招标

政府采购
├── 货物采购     ← "产品授权书"相关规则只对货物类触发
├── 服务采购
└── 工程采购

通用             ← 跨行业规则（如"投标截止日期不足20日"）
```

**为什么需要行业分类**：如果不分行业，"建筑工程施工总承包资质"的规则会在 IT 软件采购招标中触发——100% 是误报。行业分类 = 规则只对自己相关的文档类型生效。

## 思考题

1. `check: "any_match"` vs `check: "all_match"` ——什么场景下需要所有 pattern 都命中才算匹配？
2. `conditions.exclude` 和 `conditions.trigger` 是什么关系？如果 exclude 和 trigger 同时满足，谁优先？
3. 如果一条法规修订了（如 2017 版→2025 版），`source.version` 需要更新。已存在的 `source.version = "2017年修正版"` 的规则，应该保留还是修改？为什么？

## 与标书审核项目的关系

`rules/` 目录下的 YAML 文件就是 G2 组交付的"规则资产"。每条规则的 `source` 字段让审核报告可以写"经《招标投标法》第十八条检测，发现排斥性条款"——而不只是"AI 认为有问题"。
