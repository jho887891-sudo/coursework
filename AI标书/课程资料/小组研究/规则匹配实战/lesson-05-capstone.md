# Day 5：规则建模与匹配独立实验

> 前 4 天你学习了规则建模、匹配、Golden Standard 和 AI 辅助生成。今天用一组小而有代表性的规则验证完整方法，不以堆满 100 条规则作为个人门槛。

---

## 目标

搭建独立规则实验 Demo——加载 10～20 条覆盖不同边界的 YAML 规则→输入简化 ParsedDocument JSON→输出 `RuleMatch[]`，并分析至少一条误报和一条漏报。

100+ 条规则库、完整行业分类和生产接口属于小组后续项目任务或骨干选做，不作为每位成员理解课程的门槛。

---

## 架构

```
rules/*.yml                        # Git 管理的规则库（核心资产）
    │
    ▼
RuleEngine::load()                 # 加载 + 编译正则（~30 行）
    │
    ▼
RuleEngine::run(doc)               # 逐 clause 匹配（~80 行）
    │
    ▼
RuleMatch[]                        # 原始匹配结果
    │
    ▼
build_agent_context(matches, doc)  # 转化为 Agent 上下文（~30 行）
```

---

## 大作业需求

### P0：代表性规则集 + 匹配 Demo（60 分）

1. 规则集：10～20 条 YAML 规则，覆盖至少 4 个类别和正常/边界/反例
   - 排斥性条款（DISC）：≥ 20 条
   - 资质要求（QUAL）：≥ 20 条
   - 业绩要求（EXPR）：≥ 10 条
   - 报价要求（PRIC）：≥ 15 条
   - 工期要求（TIME）：≥ 10 条
   - 安全/环保（SAFE）：≥ 10 条
   - 程序合规（PROC）：≥ 15 条
2. 每条规则六要素完整 + source 追溯到法条
3. 三合一匹配引擎：输入 ParsedDocument → 输出 RuleMatch[]
4. `match_summary`：total_rules / passed / failed / uncertain / critical_failures

### P1：质量验证（20 分）

- 3 份招标文件的 Golden Standard（每份 ≥ 20 条标注）
- Recall > 95%（铁律）
- Precision > 80%
- 误报 Top-10 根因分析 + 修复

### P2：build_agent_context 集成（20 分）

- 完整的 `build_agent_context(matches, doc)` → 输出清晰的"已确认 vs 待审查"段落
- 规则库 Git 版本管理
- 单条规则启用/禁用（`enabled: true/false`）

---

## 验收标准

| 验收项 | 权重 | 怎么测 |
|--------|------|--------|
| 代表性规则六要素完整 | 25% | Schema 校验 + 口头解释 |
| Recall > 95% | 20% | 3 份 Golden Standard 验证 |
| build_agent_context 输出正确 | 15% | 输入样本→验证"已确认"和"待审查"分列清晰 |
| 匹配引擎正确输出 | 15% | 输入样本 ParsedDocument→输出 JSON→人工验证 |
| 误报分析报告 | 10% | Top-10 FP + 根因 + 修复 |
| 设计决策文档 | 15% | 规则- Agent分工/行业分类/质量保证 |

---

## 设计决策文档

1. **为什么 Recall > 95% 是铁律** — 漏检 = Agent 永远不会审这个条款。FP 的代价是 Agent 多审一个条款（几秒）——FN 的代价是废标风险逃逸。代价完全不对等
2. **为什么引擎不用复杂优化（RegexSet/分词/Pipeline）** — 100 条规则 × 500 clauses = 50000 次 `is_match`，耗时 < 1ms。在规则数量达到 10000+ 之前，简单循环足够。把时间花在规则质量上，不是引擎性能上
3. **为什么正则和关键词是互补的** — 正则捕获"句式"（"投标人在 XX 设立机构"），关键词捕获"语义"（"排斥"、"歧视"）。只用正则漏同义词表达，只用关键词漏句式变化
4. **规则引擎和 Agent 的分工边界** — 能用"是/否"回答的→规则引擎。需要语义理解的→Agent。规则引擎的产出不是终端报告——是 Agent 的上下文
