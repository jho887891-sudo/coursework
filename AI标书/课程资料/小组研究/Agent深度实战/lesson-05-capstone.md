# Day 5：🎓 独立综合实验 — 验证 Agent 框架核心机制

> 前 4 天你理解了 ReAct Loop、ToolRegistry、Coordinator、AgentBus。今天在独立 Cargo 工程中搭建最小实验环境，验证一个机制或设计判断。项目源码只用于对照，不直接修改。

---

## 目标

在课程独立目录中实现一个小型 Demo——可以是协作策略模拟器、工具可靠性实验或幻觉检测实验——用对照数据解释它在什么条件下有效、在什么条件下失败。

最低要求不是完成生产框架，而是：

1. 画出机制流程；
2. 写出可观察关键行为的最小 Demo；
3. 主动制造一个失败或退化条件；
4. 解释结果与项目架构的联系。

---

## 选题（三选一）

### 选题 A：新协作策略

**设计要求**：设计一种新的 Agent 协作策略（如动态路由——根据文档特征自动选择 Agent 组合，而非硬编码关键词匹配）。

**交付**：
1. 实现新策略的 Rust 代码（可复用现有 Coordinator 接口）
2. 在 10～20 条小型实验集上与至少 2 种基线策略做对比；50 份正式 Benchmark 和四策略全量对比为骨干选做
3. Bootstrap CI 证明新策略在某维度（F1/成本/延迟）上显著优于所有基线

**评级标准**：显著优于至少 1 个基线 → 合格。显著优于所有基线 → 优秀。

### 选题 B：新审查工具

**设计要求**：为 Agent 开发一个新的审查工具（如 `detect_boilerplate` 模板残骸识别、`compare_versions` 标书版本 Diff、`detect_contradiction` 内部矛盾检测）。

**交付**：
1. 实现 `AgentTool` trait——name/definition/execute
2. 30+ 个测试用例（正常/边界/异常），工具执行准确率 > 95%
3. 与现有工具的协作验证——Agent 能否在合适的时机正确触发该工具

**评级标准**：30 测试用例全部通过 + 工具触发验证 > 90% → 合格。

### 选题 C：幻觉检测集成

**设计要求**：将评测框架课 Day 4 的幻觉检测 Pipeline 集成到 Agent 的 `output_finding` 后处理中。

**交付**：
1. Agent 输出 `output_finding` → 拦截 → 幻觉检测器检查法规/原文/数值幻觉
2. 检测到的幻觉标记到审核报告（`hallucination_risk: true` + 具体原因）
3. 统计幻觉率（按 Agent 维度、按幻觉类型维度）

**评级标准**：假阳性率 < 20% + 至少检出 1 个真实幻觉 → 合格。

## 验收标准

| 验收项 | 权重 | 怎么测 |
|--------|------|--------|
| 功能实现 + `cargo check` | 25% | 编译 + 基本功能手动验证 |
| Benchmark 报告（含 Bootstrap CI） | 30% | 数据完整、统计分析正确、有结论 |
| 源码分析文档 | 25% | 500 字：改了什么文件、为什么这样改、效果如何 |
| 实验可解释性 | 20% | 能说明变量、现象、限制和失败原因 |

---

## 设计决策文档（必写）

1. **为什么选这个选题？** — 你的改进解决了项目的什么具体问题？
2. **为什么用这个方案而不是另一个？** — 至少对比 1 个替代方案
3. **Benchmark 数据说明了什么？** — Bootstrap CI 是否显著？效应量 d 是多少？
4. **这个改进的局限性？** — 在什么场景下你的方案会退化/不适用？
