# M2 拍照答疑 + 讲题 模块开发文档

---

## 一、模块定位与功能边界

M2 由两条业务主线组成，二者共享同一套 AI 讲解生成底座：

- **拍照答疑**：用户拍照 / 上传题目图片 → OCR 识别 → 题库匹配（命中则复用）→ AI 流式生成答案与逐步解析。
- **讲题**：基于 M1 错题本的错题，结合用户错误原因与知识水平，AI 生成"针对错误原因"的结构化逐步讲解，支持回放与追问。

两条主线最终都把讲解内容沉淀到 `teaching_contents` 表，供 M3 薄弱点分析与 M4 讲课复用。

### 1.1 功能清单

| 子功能 | 入口 | 产出 | 主要技术点 |
|--------|------|------|-----------|
| 拍照 / 上传图片 | 答疑页摄像头或相册 | 图片 URL | MediaDevices API、MinIO/OSS 存储 |
| OCR 识别 | 上传后自动触发 | 结构化题目（题干/选项/类型） | Python OCR 后处理、公式识别 |
| 题库匹配 | OCR 后自动触发 | 命中题库题或走 AI 生成 | 全文索引 / 相似度匹配 |
| 拍照答疑生成 | 用户确认题目后 | 答案 + 逐步解析 + 知识点（SSE 流式） | Prompt 工程、大模型流式回调 |
| 讲题生成 | 错题本"讲题"按钮 | 结构化讲题六段内容（SSE 流式） | 错误原因定向讲解 Prompt |
| 讲题回放管理 | 讲题历史页 | 历史讲解列表 / 详情 / 删除 | CRUD |
| 讲题追问 | 讲题页底部输入框 | 对话回复（SSE 流式） | 复用 M7 会话链路 |
| 讲题反馈 | 讲题页"懂了/不懂" | 写入用户画像事件 | 调用 M6 事件采集接口 |

### 1.2 与其他模块的协作关系

```
M1(错题本) ──→ M2(讲题)         获取要讲解的题目（GET /api/wrong-questions）
M2 ←──→ M7(即时交互)            讲题中追问走 M7 的对话链路（POST /api/conversation/ask）
M2 ──→ M6(上下文记忆)           讲题中的困惑点、反馈写入用户画像（POST /api/user-profile/{id}/record-event）
M2 ──→ M3(薄弱点)               讲题反馈数据辅助薄弱点分析（共享 teaching_contents 表）
M2 ──→ teaching_contents        讲解内容保存供复用
M2 ←── M6(用户画像)             调用 AI 前获取用户知识水平做个性化讲解
```

### 1.3 不做什么（边界约定）

- 不在本模块内自建对话引擎，追问统一走 M7 的 `POST /api/conversation/ask`。
- 不在本模块内做 TTS 朗读，语音播放调用 M8（讲题页只做播放按钮编排）。
- 不在本模块内计算薄弱点，仅把"用户反馈不懂"等事件上报给 M6。
- OCR 的底层识别能力依赖外部 OCR 服务（如百度 / 阿里 OCR API 或开源 PaddleOCR），M2 的 Python 层只做**后处理**：去噪、结构化、公式补全。

## 二、整体架构与数据流

### 2.1 技术分层

> 项目实际仓库结构为三端独立工程：`aitutor-frontend`（React + Vite + Tailwind）、`aitutor-backend-java`（Spring Boot 3 + MyBatis-Plus）、`aitutor-backend-python`（FastAPI，基于 LandPPT）。服务端口：Java 8080、Python 8000、前端 5173。数据库 `leapmind-voice`，Java 端用 Flyway 自动建表，API 文档走 Knife4j（`/doc.html`）。

| 层 | 技术栈 | 职责 | 语言 |
|----|--------|------|------|
| 前端 | React + Vite + Tailwind CSS + react-markdown + KaTeX | 摄像头/上传、OCR 结果展示、SSE 流式渲染、讲题步骤导航 | JavaScript |
| Java 后端（API 网关） | Spring Boot 3 + MyBatis-Plus + WebFlux(WebClient) + Flyway + Knife4j | 图片接收存储、SSE 透传、题库匹配、讲题内容 CRUD、JWT 鉴权限流 | Java |
| Python 后端（AI/OCR 层） | FastAPI + LangChain + 大模型 SDK（uv 管理依赖，run.py 启动） | OCR 后处理、Prompt 构建、大模型流式调用 | Python |
| 存储 | MySQL(`leapmind-voice`) + MinIO/OSS + Redis | 讲解记录、图片、缓存 | — |

### 2.2 核心调用链路

```
前端                Java(Spring Boot)              Python(FastAPI)            大模型/OCR
 │  上传图片          │                                │                          │
 │─────────────────→│ 存 MinIO → 返回 URL             │                          │
 │  OCR 识别          │                                │                          │
 │─────────────────→│ POST /api/internal/ai/ocr/process──────────────────────→│ OCR API
 │                   │←────────────── 结构化题目 ──────────────────────────────│
 │←── 结构化题目 ────│                                │                          │
 │  发起答疑(确认题目) │                                │                          │
 │  EventSource(SSE) │ 查 M6 画像 → 拼 context         │                          │
 │─────────────────→│ POST /api/internal/ai/generate/stream                    │
 │                   │──────────────────────────────→│ 构建 Prompt → 大模型流式  │
 │                   │←── SSE 流式 token ────────────│←─────────────────────────│
 │←── SSE 透传 ──────│ 聚合后写 teaching_contents      │                          │
```

### 2.3 关键设计决策

**为什么 OCR 后处理与 AI 生成放 Python？** OCR 原始结果含大量噪声（手写体干扰、公式碎片化），需要正则清洗 + 大模型二次纠错；讲题 Prompt 需要根据"用户错误原因标签"动态拼接上下文（"因为你 concept_unclear 错了，重点讲概念而非计算"）。Python 的 LangChain Prompt 模板、`str.format_map` 上下文注入比 Java 字符串拼接可靠得多。Java 只负责它擅长的事：图片上传存储、SSE 基础设施、CRUD、题库全文检索。

**为什么 SSE 而非 WebSocket？** 答疑和讲题是单向流式输出（服务器 → 前端逐 token 推送），不需要双向实时通道。SSE 基于 HTTP，自带断线重连，Spring WebFlux 的 `Flux<ServerSentEvent>` 实现成熟，足够覆盖本场景。追问场景虽是双向，但复用 M7 已有的 WebSocket/SSE 链路，M2 不重复造轮子。

## 四、数据库设计

### 4.1 新建表：ocr_records

本模块新建一张 OCR 识别记录表，用于追溯识别历史、复用识别结果（同一图片不重复调用 OCR）。

```sql
CREATE TABLE ocr_records (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id BIGINT NOT NULL COMMENT '用户ID',
  image_url VARCHAR(500) NOT NULL COMMENT '图片在OSS的URL',
  image_hash VARCHAR(64) NOT NULL COMMENT '图片内容hash，用于去重复用',
  subject VARCHAR(50) COMMENT '科目，辅助识别',
  recognized_text TEXT COMMENT 'OCR原始识别文本',
  structured_question_json TEXT COMMENT '结构化题目JSON：stem/options/type',
  confidence DECIMAL(5,3) COMMENT '识别置信度0-1',
  match_question_id BIGINT COMMENT '题库匹配命中的题目ID（NULL表示未命中）',
  match_degree DECIMAL(5,3) COMMENT '匹配度0-1',
  status VARCHAR(20) DEFAULT 'success' COMMENT 'success/failed/low_confidence',
  created_at DATETIME,
  UNIQUE KEY uk_hash (image_hash),
  INDEX idx_user_time (user_id, created_at DESC)
) COMMENT 'OCR识别记录表';
```

**image_hash 去重逻辑**：上传时计算图片 MD5/SHA256，先查 `ocr_records` 是否已有该 hash 的成功记录，命中则直接返回历史结果，避免重复消耗 OCR 配额。

### 4.2 复用共享表

本模块不重复建表，直接使用以下共享表：

| 表名 | 用途 | 读写关系 |
|------|------|---------|
| `questions` | 题库匹配、复用已有解析 | 读 |
| `knowledge_points` / `question_kp_relations` | 题目关联知识点 | 读 |
| `wrong_question_book` | 讲题来源（错题） | 读 |
| `teaching_contents` | 保存生成的答疑/讲题内容 | 写 + 读 |
| `user_profiles` | 获取用户画像做个性化 | 读 |
| `conversation_messages` | 讲题追问记录（M7 写入，M6 读取分析） | 由 M7 写 |

### 4.3 teaching_contents 在 M2 中的使用约定

`teaching_contents` 表 M4/M5 也使用，M2 写入时需约定字段取值，便于后续按类型筛选：

| 字段 | M2 写入取值 | 说明 |
|------|------------|------|
| `type` | `photo_qa`（拍照答疑）/ `explain`（讲题） | 区分两条主线 |
| `source_type` | `from_photo` / `from_wrong_question` | 来源 |
| `source_content_json` | 题目快照（stem/options/用户答案/正确答案/错误原因） | 讲题需记录用户错误答案 |
| `generated_content_json` | 完整讲解 JSON（answer/steps/knowledgePoints/tips） | 回放渲染用 |
| `ppt_structure_json` | NULL（M2 不生成 PPT） | 留给 M4/M5 |
| `status` | `published` | 讲解生成完成即发布 |

### 5.3 接口实现详解

#### 5.3.1 拍照上传与 OCR 识别

**接口**：`POST /api/ocr/recognize-question`（multipart/form-data）

**Java 编排逻辑**（`OcrService.recognizeQuestion`）：

1. 接收图片文件（`MultipartConfig` 已配置上传限制），校验格式（jpg/png）与大小（≤10MB）。
2. 计算图片 `image_hash`（SHA256），查 `ocr_records` 是否已有成功记录 → 命中直接返回。
3. 上传图片到 MinIO/OSS，得到 `image_url`。
4. 通过 `WebClient` 调用 Python `POST /api/internal/ai/ocr/process`，传 `image_url` + `subject`。
5. 接收 Python 返回的结构化题目，写 `ocr_records`。
6. 调用 `QuestionMatchService.match()` 做题库匹配，回填 `match_question_id` / `match_degree`。
7. 返回前端：OCR 结果 + 匹配信息（统一用 `ApiResponse` 包装）。若匹配度 > 0.9，附带已有解析供前端优先展示。

**Python OCR 后处理逻辑**（`ocr_processor.process`）：

1. 调用外部 OCR API 得到原始文本（含噪声）。
2. 去噪：去除页眉页脚、水印、无关标记；合并断行。
3. 结构识别：用正则区分题干与选项（`A.` `B.` 模式）；识别题型（有选项 → 选择题，否则填空/解答）。
4. 公式补全：识别碎片化公式符号（如 `a2+b2=c2` → `a²+b²=c²`），调用大模型做二次纠错（`ocr_correct.txt` 模板）。
5. 计算置信度（基于 OCR 返回的置信分 + 结构识别完整度）。
6. 返回 `{recognizedText, structuredQuestion, confidence}`。

**返回示例**：

```json
{
  "ocrRecordId": 501,
  "imageUrl": "https://oss.xxx.com/ocr/xxx.jpg",
  "recognizedText": "在直角三角形ABC中，∠C=90°，AC=3，BC=4，求AB的长。\nA. 5  B. 6  C. 7  D. 8",
  "structuredQuestion": {
```

#### 5.3.2 题库匹配服务

**接口**：`POST /api/questions/match`

**匹配策略**（`QuestionMatchService`）：

- 一级匹配：题干关键数字 + 选项指纹做 MySQL 全文索引检索（`MATCH(stem) AGAINST`）。
- 二级匹配：对候选集计算 Jaccard 相似度（题干分词后词集交集 / 并集）。
- 阈值判定：
  - `matchDegree > 0.9` → 直接返回已有解析，前端跳过 AI 生成。
  - `0.6 ≤ matchDegree ≤ 0.9` → 返回候选列表，前端弹窗让用户确认是否复用。
  - `matchDegree < 0.6` → 不命中，走 AI 生成。

```sql
-- 全文索引检索示例（questions 表需在 content_json 的 stem 上建全文索引或拆出 stem 列）
SELECT id, content_json,
       MATCH(stem) AGAINST(? IN NATURAL LANGUAGE MODE) AS score
FROM questions
WHERE subject = ? AND type = ?
ORDER BY score DESC
LIMIT 5;
```

> 实现提示：`questions.content_json` 是 JSON 文本，建议新增 `stem_text` 生成列并建全文索引，避免每次 JSON_EXTRACT 影响性能。

#### 5.3.3 拍照答疑生成（SSE 流式）

**接口**：`POST /api/explain/photo-qa`，返回 `text/event-stream`

**Java 编排逻辑**（`PhotoQaService.generatePhotoQa`）：

1. 入参：`userId`、`ocrRecordId`、`question`（结构化题目）、可选 `userProfile`。

5. 透传 Python 的 SSE 给前端；同时用 `StringBuilder` 聚合完整内容。
6. 流结束时（收到 `done` 事件），把聚合内容写入 `teaching_contents`（`type=photo_qa`、`source_type=from_photo`），返回 `contentId`。

#### 5.3.4 讲题生成（SSE 流式）

**接口**：`POST /api/explain/generate`，返回 `text/event-stream`

讲题与拍照答疑的区别在于：讲题**针对用户具体错误原因**讲解，输入是 M1 错题本的错题，而非 OCR 题目。

### 5.4 SSE 流式实现要点

Java 端使用 Spring WebFlux 返回 `Flux<ServerSentEvent>`。代码位于 `controller/lesson/PhotoQaController`，调 Python 的部分参考已有 `service/lesson/AIModelService` 通过 `WebClient`（`WebClientConfig` 注入）流式调用的写法：

```java
package com.treepeople.leapmindtts.controller.lesson;

@RestController
public class PhotoQaController {

    private final PhotoQaService photoQaService;

    @PostMapping(value = "/api/explain/photo-qa", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<ServerSentEvent<String>> photoQa(@RequestBody PhotoQaRequestDTO req) {
        // PhotoQaService 内部：WebClient 调 M6 取画像 → WebClient 调 Python 流式 → 透传 + 聚合落库
        return photoQaService.generatePhotoQa(req);
    }
}
```

`PhotoQaService.generatePhotoQa` 核心逻辑（聚合与透传）：

```java
public Flux<ServerSentEvent<String>> generatePhotoQa(PhotoQaRequestDTO req) {
    // 1. WebClient 调 M6 取画像
    UserProfileSummary profile = fetchProfile(req.getUserId());
    // 2. 组装发往 Python 的请求体
    AICallRequest aiReq = buildPhotoQaRequest(req, profile);
    // 3. WebClient 调 Python 流式接口（参考 AIModelService 写法）
    StringBuilder aggregator = new StringBuilder();

    return webClient.post()
        .uri("/api/internal/ai/generate/stream")
        .bodyValue(aiReq)
        .retrieve()
        .bodyToFlux(String.class)              // 逐行读取 Python SSE
        .map(chunk -> {
            SseEventDTO event = parseSseChunk(chunk);
            if (event.getContent() != null) aggregator.append(event.getContent());
            return ServerSentEvent.<String>builder().data(toJson(event)).build();
        })
        .concatWith(Mono.defer(() -> {
            // 流结束：落库 + 返回 done 事件
            Long contentId = saveTeachingContent(req, aggregator.toString());
            return Mono.just(ServerSentEvent.<String>builder()
                .data(toJson(new SseEventDTO("done", null, contentId))).build());
        }))
        .onErrorResume(e -> Mono.just(ServerSentEvent.<String>builder()
            .data(toJson(new SseEventDTO("error", "生成失败：" + e.getMessage(), null))).build()));
}
```

关键点：
- 透传与聚合并行：每个 chunk 既推给前端又追加到 `aggregator`，流结束后一次性落库。
- 错误降级：捕获异常后发 `error` 事件，前端展示兜底提示，不直接断流。
- 超时熔断：单次 AI 调用超过 30 秒自动中断（WebFlux `timeout`）。
- 限流：复用 M7 的 `ai:ratelimit:{userId}` 机制，单用户每分钟最多 10 次讲解生成。

### 5.5 Java↔Python 内部通信契约

M2 调用的 Python 内部接口（均通过 `WebClient` 调用，参考 `service/lesson/AIModelService` 现有实现）：

| Python 接口 | 方法 | M2 用途 | 返回 |
|-------------|------|---------|------|
| `/api/internal/ai/ocr/process` | POST | OCR 后处理 | JSON |
| `/api/internal/ai/generate/stream` | POST | 答疑/讲题流式生成 | SSE |
| `/api/internal/ai/generate` | POST | 非流式（备用，如 OCR 纠错） | JSON |

`WebClient` 调用封装要点（建议在 `AIModelService` 中扩展或新增工具方法）：
- 非流式调用：`webClient.post().uri(...).retrieve().bodyToMono(AICallResult.class)`。
- 流式调用：`webClient.post().uri(...).retrieve().bodyToFlux(String.class)`，逐行读取 Python 的 SSE。
- 重试：失败自动重试 3 次，指数退避（1s/3s/9s）。
- 超时：连接 5s，读取 30s。
- 基地址与超时由 `WebClientConfig` 统一配置，指向 `http://localhost:8000`。

## 十、验收标准

| 验收项 | 标准 |
|--------|------|
| 拍照识别准确率 | 印刷体题目识别率 > 95%，支持数学公式还原 |
| AI 解答质量 | 解答步骤清晰、知识点标注正确、结论正确 |
| 流式讲题体验 | SSE 流式返回，首字延迟 < 2 秒 |
| 错题讲解个性化 | 讲解针对用户具体错误原因，而非通用答案（4 种 wrongReasonTag 产出差异化讲解） |
| 讲解可回放 | 历史讲解可重新查看，内容完整、公式渲染正常 |
| 题库匹配优先 | 相同题目命中后不重复调用 AI |
| 同图去重 | 相同图片重复上传不重复消耗 OCR 配额 |
| 追问链路 | 讲题追问走 M7，对话流式正常，记录可被 M6 采集 |
| 反馈闭环 | "不懂"反馈可写入用户画像 confusion_history |
| 限流降级 | 单用户超频限流，AI 超时熔断有兜底提示 |

## 十一、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| OCR 对手写体识别率低 | 答疑体验差 | 大模型二次纠错 + 前端允许手动修改识别结果 |
| 数学公式 LaTeX 渲染失败 | 公式乱码 | KaTeX 渲染失败时降级为纯文本展示，并在 Prompt 中约束输出标准 LaTeX |
| 大模型流式输出格式不稳定 | 前端解析失败 | SSE chunk 用 JSON 包装 type 字段，前端按 type 容错解析；Python 端约束模型输出结构 |
| 题库全文索引性能差 | 匹配慢 | stem 拆出独立列建全文索引，候选集限定科目+题型缩小范围 |
| M1/M6/M7 接口未就绪 | 联调阻塞 | 早期用 Mock 数据并行开发，定义接口契约后桩替换 |
