## 1. 文档说明

### 1.1 编写目的

本文档定义 LeapMind 教育网站 M6“上下文记忆（全局）”模块的业务边界、功能需求、数据模型、接口契约、技术分工、非功能指标、测试与验收标准，用于指导前端、Java 后端、Python 画像服务及 M1～M8 相关模块的开发和联调。

### 1.2 模块定位

M6 是全站个性化学习的上下文基础设施，负责完成三件事：

1. 统一接收各模块产生的学习事件；
2. 将事件计算为可解释、可更新、可控制的用户学习画像；
3. 按调用场景向 AI 模块提供最小必要上下文。

M6 不负责题库、讲题内容生成、即时讲课或通用对话本身。来源模块仍拥有原始业务数据；M6 保存标准化事件和画像派生结果，不复制无必要的业务原文。

### 1.3 需求依据

- 《LeapMind 教育网站——模块详细任务文档》V2.0；
- 仓库中的模块文档结构与接口命名约定；
- 项目技术栈：React + TypeScript、Spring Boot、Python FastAPI、MySQL、Redis；
- Java 与 Python 通过内网 HTTP REST 通信，Python 服务基地址按项目总文档统一为 `http://localhost:8001`。

### 1.4 术语

| 术语 | 定义 |
|---|---|
| 学习事件 | 用户在业务模块发生的一次可追踪学习行为 |
| 会话上下文 | 当前会话中用于连续问答的消息、引用对象和摘要 |
| 阶段记忆 | 最近 7～30 天的关注点、困惑和任务状态 |
| 长期画像 | 跨会话维护的知识掌握趋势、稳定偏好和复习状态 |
| 场景摘要 | 从完整画像中按 `sceneType` 裁剪出的 AI 上下文 |
| 掌握度 | 某知识点的综合学习状态，取值 0～1 |
| 置信度 | 系统对某画像推断的可信程度，取值 0～1 |
| 证据 | 支撑画像结论的事件类型、样本量、时间窗口和来源模块 |

## 2. 建设目标与边界

### 2.1 建设目标

| 编号 | 目标 |
|---|---|
| G-01 | 建立统一、幂等、可版本化的学习事件入口 |
| G-02 | 在关键事件产生后 5 分钟内更新相关画像 |
| G-03 | 为 M2、M4、M5、M7 提供统一场景摘要接口 |
| G-04 | 形成学习、讲解、反馈、复习的闭环 |
| G-05 | 保证画像可解释、可纠正、可删除、可关闭 |
| G-06 | M6 故障时不阻断其他模块的核心学习流程 |

### 2.2 本期范围

- M1、M2、M3、M4、M7 关键事件接入，M5 可选反馈接入；
- 单条/批量采集、字段校验、幂等去重、失败记录；
- 知识掌握度、近期困惑、最近关注点、可观察讲解偏好、学习节奏计算；
- 完整画像、场景摘要、知识掌握状态和证据查询；
- 上下文裁剪、权限检查、Prompt 注入和无画像降级；
- 1/3/7/14/30 天复习计划与站内提醒；
- 学习档案、复习提醒和记忆设置前端；
- 缓存、审计、保留期和删除机制。

### 2.3 非本期范围

- 心理、医学、智力或人格诊断；
- 将用户固定归类为“视觉型/听觉型学习者”；
- 跨学校个人画像共享；
- 自动分班、奖惩、升学判断等高影响决策；
- 深度学习推荐模型和实时特征平台；
- 短信、邮件、微信等站外提醒。

### 2.4 角色与权限

| 角色 | 可执行操作 | 限制 |
|---|---|---|
| 学生 | 查看本人画像/证据/复习计划，修改偏好，删除或关闭记忆 | 不可访问他人数据 |
| 教师 | 查看授权学生或班级的教学相关画像 | 默认不显示完整对话原文 |
| 监护人 | 经授权查看趋势与复习提醒 | 遵循最小必要原则 |
| 管理员 | 配置事件类型、保留期，处理数据请求 | 所有操作记录审计日志 |
| 业务服务 | 按场景获取摘要、上报事件 | 不得直接查询其他场景数据或底层全量事件 |

## 3. 总体设计

### 3.1 上下文分层

| 层级 | 内容 | 生命周期 | 主要责任模块 |
|---|---|---|---|
| 会话上下文 | 最近消息、当前题目/课程引用、会话摘要 | 会话结束后按策略过期 | M7 管理，M6 只接收结构化结果 |
| 阶段记忆 | 近期困惑、最近关注点、当前复习任务 | 7～30 天滚动更新 | M6 |
| 长期画像 | 掌握趋势、稳定偏好、长期复习状态 | 用户启用期间持续维护 | M6 |

单次错误或临时情绪不得直接形成长期画像。只有样本量、时间跨度和置信度达到阈值后，才可形成稳定结论。

### 3.2 模块关系

```text
M1 答题事件 ──────┐
M2 讲题反馈 ──────┤
M3 薄弱点变化 ────┤
M4 讲课互动 ──────┼──> Java 事件采集 ──> MySQL user_events
M5 内容使用反馈 ──┤                          │
M7 问答困惑 ──────┘                          v
                                      Python 画像计算
                                             │
                       ┌─────────────────────┴──────────────────┐
                       v                                        v
              用户画像/复习计划                         场景化摘要接口
                       │                                        │
                       └────────> M2 / M4 / M5 / M7 <───────────┘
```

### 3.3 Java 与 Python 分工

| 层 | 职责 | 关键技术 |
|---|---|---|
| React 前端 | 学习档案、知识掌握详情、复习提醒、记忆设置 | React + TypeScript |
| Java API 层 | 鉴权、事件采集、幂等、查询、缓存、权限、审计、任务编排 | Spring Boot + MyBatis-Plus + Redis |
| Python 画像层 | 多源聚合、掌握度计算、困惑提取、偏好趋势、画像摘要 | FastAPI + pandas/numpy + jieba/规则 |
| 数据层 | 原始事件、派生画像、复习计划、设置、审计 | MySQL + Redis |

首版优先使用可解释规则和加权模型，不引入 KMeans 等难以解释且样本不足的聚类方案。Python 负责计算，Java 负责业务边界与对外契约。

### 3.4 主流程

1. 来源模块生成全局唯一 `eventId`，按统一 schema 上报。
2. Java 校验登录身份、来源模块、字段、事件类型和 schema 版本。
3. Java 以 `eventId` 幂等写入 `user_events`，返回 `ACCEPTED`。
4. 关键事件触发增量画像任务；普通事件按小时聚合。
5. Python 读取/接收所需事件，计算画像项并返回算法版本与证据摘要。
6. Java 在事务中写入画像、知识点状态和复习计划，随后失效 Redis 缓存。
7. M2/M4/M5/M7 调用摘要接口时，Java 完成权限、场景裁剪和长度限制。
8. 调用方将摘要作为“不可信参考数据”传入 Python AI 层；当前用户指令优先。
9. 若 M6 超时或无数据，返回 `NO_PROFILE` 或调用方降级为通用讲解。

### 3.5 更新策略

- `answer_question`、`explanation_feedback=still_confused`、`ask_doubt`：5 分钟内增量更新；
- 播放、暂停、浏览等高频事件：按小时聚合；
- 每天凌晨执行全量校准；
- 画像每次更新保存 `algorithmVersion`、`evidenceCount`、`windowStart`、`windowEnd` 和 `confidence`；
- 新用户或证据不足时返回 `INSUFFICIENT_EVIDENCE`；
- 用户手动偏好覆盖算法推断，用户可恢复“自动判断”。

## 4. 功能需求

### FR-01 事件采集

#### FR-01-01 公共事件结构

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `eventId` | string(64) | 是 | 来源模块生成的全局唯一 ID |
| `userId` | long | 是 | 必须与路径及身份一致 |
| `eventType` | string | 是 | 注册的标准事件类型 |
| `sourceModule` | string | 是 | M1/M2/M3/M4/M5/M7/M6 |
| `occurredAt` | datetime | 是 | 实际发生时间，ISO 8601 |
| `schemaVersion` | string | 是 | 初始为 `1.0` |
| `sessionId` | string | 否 | 学习或对话会话 ID |
| `kpId` | long | 否 | 关联知识点 ID |
| `traceId` | string | 否 | 跨服务追踪 ID |
| `data` | object | 是 | 事件扩展字段 |

#### FR-01-02 事件目录

| 事件类型 | 来源 | 最小扩展字段 |
|---|---|---|
| `answer_question` | M1 | `isCorrect`、`difficulty`、`timeSpentSec`、`hintCount` |
| `finish_practice` | M1 | `questionCount`、`accuracy`、`durationSec` |
| `request_explanation` | M2 | `explainId`、`reasonTag` |
| `explanation_feedback` | M2 | `explainId`、`feedback`、`repeatCount` |
| `weak_point_changed` | M3 | `oldScore`、`newScore`、`reason` |
| `lecture_interact` | M4 | `lectureId`、`chapterId`、`action` |
| `lesson_material_used` | M5 | `contentId`、`materialType`、`result` |
| `ask_doubt` | M7 | `topic`、`confusionTag`、`isFollowUp` |
| `mark_reviewed` | M6 | `result`、`timeSpentSec`、`hintCount` |
| `preference_changed` | M6 | `preferenceKey`、`preferenceValue` |

#### FR-01-03 规则

- 相同 `eventId` 只落库和消费一次；重复请求返回首次处理结果。
- 批量接口最多 100 条，采用逐条结果，禁止因一条失败回滚全部成功项。
- 未知事件类型返回 `EVENT_TYPE_UNSUPPORTED`；未知 schema 返回 `SCHEMA_VERSION_UNSUPPORTED`。
- `occurredAt` 超出服务器时间前后 24 小时的事件进入异常队列，不直接参与画像计算。
- 不保存密码、访问令牌、身份证号、精确位置等无关敏感数据。
- 对话默认仅保存知识点、意图和困惑标签；原文保存需要独立授权和保留期。

### FR-02 画像计算

#### FR-02-01 知识掌握度

MVP 使用可配置加权模型：

```text
masteryScore = 0.35 × accuracyScore
             + 0.20 × difficultyScore
             + 0.15 × stabilityScore
             + 0.15 × independenceScore
             + 0.15 × recencyScore
```

所有子项归一化到 0～1，权重之和必须为 1，并保存算法版本。状态映射：

| 分数 | 状态 |
|---|---|
| 0～0.39 | 薄弱 |
| 0.40～0.59 | 待巩固 |
| 0.60～0.79 | 基本掌握 |
| 0.80～1.00 | 熟练 |

有效样本少于 5 次时，状态为 `INSUFFICIENT_EVIDENCE`，不进入强弱项列表。

#### FR-02-02 近期困惑

系统综合连续答错、重复讲解、反馈“仍不理解”、重复追问、同一章节频繁回看及 M3 薄弱点变化。每个困惑项包含：`kpId`、`detail`、`frequency`、`lastOccurredAt`、`evidenceTypes`、`confidence`。默认窗口 30 天，旧证据时间衰减。

#### FR-02-03 可观察偏好

系统只描述可观察的内容偏好，如 `diagram`、`step_by_step`、`worked_example`、`concise`、`detailed`、`audio`。不把用户固定标记为某种“学习者类型”。

- 推断偏好必须附置信度；
- 用户主动设置优先；
- 数据不足使用系统默认值；
- 每 30 天重新评估推断偏好。

为兼容原始总文档中的 `learningStyle` 字段，对外响应可保留该字段但标记为 `deprecated`，新调用方使用 `preferredContentModes` 与 `preferredExplanationStyle`。

#### FR-02-04 学习节奏与关注点

- 学习节奏结合用户历史中位数和同年级同难度合理区间，输出 `slow/moderate/fast`；前端显示“偏细致/适中/偏快速”。
- 最近关注点取最近 7 天有效学习行为的前三个知识点，单纯浏览不计为强信号。

### FR-03 画像查询与证据

系统提供完整画像、场景摘要、知识掌握状态和画像证据查询。完整画像只供本人学习档案和授权教师端；其他模块只能通过场景摘要接口获取最小数据。

数据不足时必须返回状态与原因，禁止以空数组掩盖计算尚未完成，也禁止让大模型虚构画像。

### FR-04 场景摘要与 AI 注入

| `sceneType` | 消费方 | 允许字段 | 禁止字段 |
|---|---|---|---|
| `explaining` | M2 | 当前知识点状态、错误趋势、讲解偏好 | 无关学科、完整对话原文 |
| `lecturing` | M4 | 课程相关薄弱点、近期关注、内容偏好 | 无关私人信息 |
| `lesson_prep` | M5 | 授权班级的聚合画像 | 未授权单个学生画像 |
| `conversation` | M7 | 当前会话摘要、相关困惑、主动偏好 | 无关历史和高敏感字段 |

规则：

1. 当前用户明确要求优先于历史偏好。
2. 摘要按相关性排序并限制长度，默认不超过 1200 个中文字符。
3. 历史内容使用独立的数据区块传递，不与 system prompt 混写。
4. 每次注入记录模块、场景、画像版本、字段集合和时间。
5. M6 超时、禁用或无数据时返回可识别状态，调用方继续通用模式。

### FR-05 复习计划

MVP 采用 1、3、7、14、30 天阶段间隔：

| 复习结果 | 阶段变化 |
|---|---|
| 独立正确 | 进入下一阶段 |
| 正确但使用提示 | 保持当前阶段 |
| 错误或仍不理解 | 回到第一阶段 |
| 主动推迟 | 重排时间，不修改掌握结论 |

同一知识点每天最多一次站内提醒。用户可关闭提醒并设置免打扰时段。

### FR-06 学习档案前端

#### 页面和路由

| 路由 | 页面 | 内容 |
|---|---|---|
| `/learning-profile` | 学习档案 | 摘要、知识状态、趋势、近期困惑 |
| `/learning-profile/knowledge/:kpId` | 知识点详情 | 掌握度、证据、复习计划、历史趋势 |
| `/learning-profile/reviews` | 复习提醒 | 今日、即将到期、已逾期 |
| `/settings/memory` | 记忆设置 | 开关、偏好、删除、数据用途说明 |

#### 组件

`ProfileSummary`、`KnowledgeTree`、`MasteryProgress`、`EvidenceDrawer`、`RecentConfusions`、`ReviewReminderCard`、`MemorySettingsForm`。

前端不得只用颜色表示掌握状态；所有图表应提供文字标签和键盘可操作入口。

### FR-07 用户控制

用户可以：

- 查看系统保存的主要长期记忆及用途；
- 修正年级、讲解方式等可编辑字段；
- 删除单项记忆或全部长期画像；
- 关闭长期记忆、AI 个性化注入、原文使用和复习提醒；
- 查看删除任务状态。

关闭长期记忆后仍可保留完成当前请求所需的短期会话上下文；关闭个性化注入后，业务模块不得获取长期画像。

### FR-08 权限、审计与降级

- 所有接口执行身份和资源权限校验；
- 服务间调用使用受限服务身份，禁止依赖前端传入角色；
- 画像查看、导出、修改、删除和管理员操作写入审计日志；
- 日志不记录不必要的对话原文和完整画像；
- Python 计算失败时保留上一版有效画像并标记 `STALE`；
- Redis 不可用时回源 MySQL；MySQL 写入失败时事件返回失败，不得假成功；
- M6 不可用时其他 AI 模块无画像降级。

```json
  "result": "CORRECT_WITHOUT_HINT",
  "timeSpentSec": 80,
  "occurredAt": "2026-07-18T20:00:00+08:00"
}
```

### 5.5 统一错误响应

```json
{
  "code": "SCHEMA_VERSION_UNSUPPORTED",
  "message": "event schema version 2.0 is not supported",
  "details": [{"field": "schemaVersion", "reason": "supported: 1.0"}],
  "requestId": "req_2d01",
  "timestamp": "2026-07-18T10:20:30+08:00"
}
```

核心错误码：`INVALID_ARGUMENT`、`UNAUTHENTICATED`、`FORBIDDEN`、`EVENT_TYPE_UNSUPPORTED`、`SCHEMA_VERSION_UNSUPPORTED`、`RATE_LIMITED`、`PROFILE_NOT_READY`、`INTERNAL_ERROR`。

### 5.6 Python 内部接口

```http
POST /api/internal/ai/build-profile
```

请求只包含计算所需的结构化事件或数据引用；返回画像项、证据摘要、算法版本、置信度和更新时间。该接口仅允许 Java 服务访问，不直接暴露给前端。

如 M7 需要压缩长会话，可复用项目约定的：

```http
POST /api/internal/ai/compress-context
```

M7 负责会话消息生命周期，M6 只接收压缩后的困惑标签和相关知识点，避免职责重叠。

### 6.6 审计与缓存

- `profile_audit_logs` 记录查看、导出、修改、删除、配置变化，不记录完整敏感内容；
- Redis 画像缓存：`user:profile:{userId}`，TTL 30 分钟；
- 场景摘要缓存：`user:profile:summary:{userId}:{sceneType}:{kpId}`，TTL 10 分钟；
- 复习提醒：`user:review:due:{userId}:{yyyyMMdd}`，TTL 至次日；
- 画像写入成功后，通过用户维度索引失效相关缓存，禁止仅等待 TTL。

## 7. 非功能需求

### 7.1 性能与容量

| 指标 | 目标 |
|---|---|
| 单条事件接收 P95 | < 200ms |
| 批量 100 条事件接收 P95 | < 1s |
| 场景摘要缓存命中 P95 | < 300ms |
| 场景摘要缓存未命中 P95 | < 800ms |
| 关键事件画像更新时间 | 95% < 5 分钟 |
| 学习档案首屏 API P95 | < 1s |

首版容量按 1 万日活、每用户每天 100 条事件估算；事件表按月归档预留方案。

### 7.2 可用性与一致性

- M6 月度可用性目标不低于 99.5%；
- 事件采用至少一次投递，消费端通过 `eventId` 幂等；
- 画像明细、摘要版本与缓存失效必须在同一更新流程中完成；
- Python 失败时不得覆盖上一版有效画像；
- 调用方必须配置超时和降级，M6 失败不阻断主流程。

### 7.3 安全与隐私

- 所有外部接口必须认证，资源访问必须校验用户或授权关系；
- 服务间调用使用独立凭证和最小权限；
- 传输使用 TLS，数据库敏感字段按项目规范加密或脱敏；
- 明示采集目的、字段和保留期；
- 支持访问、更正、删除、关闭和导出请求；
- 画像结论不得包含侮辱性、歧视性或医学化表述。

### 7.4 可解释与可维护

- 每项画像必须能追踪算法版本、时间窗口、证据类型、样本数和置信度；
- 阈值、权重、保留期和复习间隔配置化；
- 事件 schema 和画像 schema 版本化；
- Java/Python 契约通过 OpenAPI 与契约测试固定；
- 全链路携带 `traceId`，监控采集成功率、计算延迟、摘要延迟、降级率和缓存命中率。

## 8. 测试与验收

### 8.1 关键测试用例

| ID | 场景 | 前置/输入 | 预期结果 |
|---|---|---|---|
| AT-01 | 幂等采集 | 相同 `eventId` 连续提交两次 | 仅一条记录，第二次返回原结果 |
| AT-02 | 身份不一致 | 路径用户与请求体用户不同 | 返回 `FORBIDDEN`，不写库 |
| AT-03 | 版本不支持 | `schemaVersion=2.0` | 明确错误，不静默丢弃 |
| AT-04 | 关键更新 | 答错 + `still_confused` | 5 分钟内更新相关困惑 |
| AT-05 | 数据不足 | 某知识点仅 2 条样本 | 返回 `INSUFFICIENT_EVIDENCE` |
| AT-06 | 场景裁剪 | M2 查询 `kpId=10` | 不返回无关学科或完整对话 |
| AT-07 | 当前指令优先 | 历史偏好“简洁”，当前要求“详细” | 使用详细讲法 |
| AT-08 | 注入关闭 | 关闭 personalization | 摘要返回 `DISABLED`，AI 走通用模式 |
| AT-09 | Python 失败 | 画像计算超时 | 保留旧版本并标记 `STALE` |
| AT-10 | Redis 失败 | 缓存不可用 | 回源 MySQL，功能仍可用 |
| AT-11 | 复习推进 | 独立答对 | 进入下一阶段并更新日期 |
| AT-12 | 删除记忆 | 删除单项偏好 | 数据、派生结果、缓存均不可再查 |
| AT-13 | 无画像降级 | M6 服务不可用 | M2/M4/M5/M7 主流程正常 |
| AT-14 | 可访问性 | 键盘操作学习档案页 | 可聚焦、可读、非纯颜色表达 |

### 8.2 验收标准

1. M1、M2、M3、M4、M7 每类至少一种关键事件完成端到端接入。
2. 单条/批量事件具备校验、幂等、版本错误和部分失败处理。
3. 画像包含知识掌握、近期困惑、关注点、偏好、节奏和复习状态，并附置信度与版本。
4. M2、M4、M5、M7 可通过同一摘要接口按场景获取数据。
5. 同一问题在有/无相关画像时可观察到合理讲解差异，且事实答案不受画像改变。
6. 关键事件 95% 在 5 分钟内反映到画像。
7. 学习档案、知识点详情、复习提醒、记忆设置四个页面可用。
8. 用户可查看依据、修正偏好、关闭注入、删除记忆。
9. M6、Python 或 Redis 单点异常均有明确降级路径。
10. AT-01～AT-14 全部通过，P0/P1 缺陷为 0。

## 12. 结论

M6 的核心价值是把“历史数据”转化为“当前场景可用且用户可控制的学习上下文”。本方案在项目原始 M6 设计上补齐事件幂等、schema 版本、场景裁剪、画像证据、用户控制、降级和可测验收，同时保持 React、Spring Boot、FastAPI、MySQL、Redis 的既定技术路线，适合作为开发、联调和提交评审的统一依据。
